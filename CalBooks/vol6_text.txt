4. Authorization Model
4.1 Permission Structure
Permissions are expressed as module:action pairs (e.g. receivables:write, ledger:read), matching the bounded-context/API-resource ownership fixed in Vol. 2 Section 11. Every API endpoint declares its required permission(s) at the controller level (Vol. 3 Section 4), and the Gateway/middleware layer rejects any request whose JWT permissions do not include it - authorization is enforced centrally, not re-implemented per module.
4.2 Custom Roles (Vol. 1 BC-17) - Ceiling Rule
Tenant administrators may define custom roles (Enterprise edition, Vol. 1 Section 9) by composing permissions, but a custom role can never grant a permission the composing admin does not themselves hold, and no custom role can exceed the permission ceiling of the closest base role it resembles. This prevents the privilege-escalation threat identified in Section 2.2 where a misconfigured custom role could otherwise grant unintended access.
4.3 Resource-Level Authorization
Role-based permissions grant module-level access; resource-level checks (e.g. "can this Sales Staff user view this specific customer's invoices" under a tenant that restricts by sales territory) are a Release 2/3 capability layered on top of the base RBAC model, implemented as an additional policy check in the domain service layer (Vol. 2 Section 6.1 component pattern), not by modifying the JWT/role model itself.

5. Data Protection & Encryption
5.1 Encryption in Transit
TLS 1.2+ enforced on every external-facing endpoint (web app, API gateway, portal, public API) - no plaintext HTTP, including internal health-check endpoints exposed beyond the deployment network.

6. Secrets & Credential Management
6.1 Key Management
A managed Key Management Service (cloud-provider KMS, per Vol. 3 Section 2.1's eventual cloud-provider confirmation) holds the master keys for the envelope-encryption pattern in Section 5.2. Application code never handles a master key directly - it calls the KMS to wrap/unwrap DEKs, and master key rotation is a KMS-native operation that does not require re-encrypting all data at rest.
6.2 Secrets Store (formalizing Vol. 5 Section 9's Credential Store)
6.3 Secrets Hygiene Rules
Secrets are never written to application logs (Vol.3 Section 6) - log statements are code-reviewed and CI-scanned for accidental secret interpolation.
Secrets are never committed to the monorepo (Vol.3 Section 3.1) - a pre-commit and CI secret-scanning gate blocks any match against common credential patterns.
Every credential has a defined owner and rotation cadence, tracked in an internal secrets inventory (Vol.7 operational runbook).

7.2 Network Segmentation
The database, cache, and event bus (Vol. 3 Section 2) are not directly reachable from outside the private network - only the API Gateway and background workers can reach them. 9. Audit Logging & Tamper-Evidence
Formalizes Vol. 1 BC-20's audit trail requirement and Vol. 1 Principle P4's auditability principle into a concrete design.
9.1 What Is Audited
Every financial-entity state change (invoice sent/voided, journal posted, bill approved) - already captured as domain events (Vol.2 Section 10) and additionally written to a dedicated append-only audit_log table.
Every authentication event (login success/failure, MFA challenge, password reset, role change) - security-relevant even where no financial entity changed.
Every permission/role configuration change (Section 4.2) - who granted what to whom, and when.
Every access to a decrypted secret (Section 6.2) - logged with the requesting service identity, not just the tenant.
9.2 Tamper-Evidence Mechanism
Audit log entry - hash-chained for tamper evidence

Each audit entry includes a hash of the previous entry, forming a per-tenant hash chain; any retroactive edit or deletion of a past entry breaks the chain and is detectable by a periodic integrity-verification job. This does not require exotic infrastructure (e.g. a blockchain) - the hash chain plus the append-only database constraint (Vol.4 Section 2, D2/D6) is sufficient tamper-evidence for the compliance bar this platform targets (Section 10).
