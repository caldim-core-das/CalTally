1. Introduction & Scope
1.1 Purpose
This volume turns the domain model (Vol. 2 Section 8 UML), the bounded-context ownership map (Vol. 2 Section 3.2), and the PostgreSQL/shared-schema/RLS decision (Vol. 3 Section 2.2, 11) into a physical data architecture: entity-relationship diagrams per bounded context, physical table definitions for the highest-traffic entities, indexing and partitioning strategy, the concrete Row-Level Security implementation, and the schema migration and data retention strategy.
1.2 Schema Ownership Rule
Consistent with Vol. 3 Section 3 ("a module may only import another module's public export surface"), the same rule applies at the data layer: each bounded context owns a PostgreSQL schema (namespace) and is the only code allowed to write to its own tables. Other contexts read via the owning context's application-layer API (Vol. 2 Section 11), never via direct SQL joins across schema boundaries. Where a report genuinely needs cross-context joins, it queries the OpenSearch read-model (Vol. 3 Section 2.6) built from published domain events, not the transactional schemas directly.

5. Indexing Strategy
General rule: tenant_id is always the leading column of every composite index, since RLS means every query implicitly filters on it - a non-leading tenant_id would force the planner to scan across tenants before filtering.

6. Partitioning Strategy
Only tables with unbounded per-tenant growth and high aggregate write volume across all tenants are partitioned; small reference tables are not, to avoid unnecessary operational complexity.
Explicitly NOT partitioned at Release 1: identity_tenant.tenants, ledger.accounts, receivables.customers, inventory.items - these are reference/master data with bounded per-tenant row counts (Section 11 classification).

7. Row-Level Security Implementation

9. Migration Strategy
9.1 Tooling & Process
Schema migrations are managed with a code-first migration tool (e.g. node-pg-migrate or Prisma Migrate, consistent with the TypeScript stack in Vol. 3) with every migration checked into the same monorepo as the module it belongs to (Vol. 3 Section 3.2).
Because the platform uses a single shared schema per environment (not per-tenant databases, Vol. 3 Section 11), a migration runs once per environment and applies to all tenants simultaneously - this makes migration safety a first-class operational concern, not an edge case.
All migrations must be backward-compatible with the currently-deployed application version for at least one release cycle (expand/contract pattern): add new columns as nullable or with defaults first, deploy code that writes to both old and new shapes, backfill, then drop the old column in a later migration.
Destructive migrations (DROP COLUMN, DROP TABLE) require a two-step process: mark deprecated in one release, physically remove in the next, with a manual sign-off gate (Vol. 8 CI pipeline).

9.3 Large-Table Migration Safety (journal_lines, invoice_line_items)
Because these tables are partitioned (Section 6), schema changes are applied per-partition in small batches rather than as a single blocking ALTER TABLE across the entire (potentially billions-of-rows) table - avoiding long-held locks that would stall the append-only ledger writes financial correctness depends on.
