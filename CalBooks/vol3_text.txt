6. Logging Strategy
6.1 Principles
Structured JSON logging only - no unstructured string logs - so logs are queryable in the observability platform (Vol. 7 will select the concrete tool, e.g. an ELK/OpenSearch or vendor SaaS stack).
Every log line carries: tenantId, userId (if applicable), correlationId, module (CTX-xx), and a severity level.
Correlation ID is generated at the API Gateway (Vol. 2 Section 5) on every inbound request and propagated through every internal call and published event, so one user action can be traced end-to-end across modules even inside the monolith.
Financial-mutation log lines (ledger postings, invoice status changes) are never sampled or dropped - they log at INFO level unconditionally, distinct from high-volume debug logs which may be sampled under load.

7. Caching Strategy
7.1 Cache Layers
7.2 Cache Key Convention (Tenant-Safe)
Cache key pattern - tenant ID is always the first segment

No cache key is ever constructed without a tenant ID prefix - this is a CI-enforced lint rule (Section 4) to prevent the single highest-risk multi-tenancy bug class: cross-tenant cache leakage. A shared cache utility wraps all Redis access so individual modules cannot bypass the key convention.

8. Queue & Asynchronous Processing Strategy
8.1 Event Bus Topic Design
Kafka topics are partitioned by tenantId + aggregateId to guarantee per-aggregate ordering (e.g. all events for one invoice arrive in order) while allowing horizontal consumer scaling across tenants.
8.2 Delivery Semantics & Idempotent Consumers
At-least-once delivery is assumed for all Kafka consumers (exactly-once is not guaranteed end-to-end across this stack). Every consumer must be idempotent - reprocessing the same event must not double-post a journal entry or double-send a notification.
Idempotency is achieved by consumers tracking a processed-event-id ledger (keyed by event ID, checked before applying side effects) rather than relying on the broker alone.
Poison messages (repeatedly failing to process) are routed to a dead-letter topic after a bounded retry count (e.g. 5 attempts with exponential backoff) and surfaced as an operational alert (Vol. 7).
