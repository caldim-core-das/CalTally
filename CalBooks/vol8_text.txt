1. Introduction & Scope
1.1 Purpose
This volume defines the complete test strategy needed to ship the platform with confidence: the test pyramid and ownership model, concrete testing standards for each layer (unit, integration, contract, end-to-end), the formalized security test plan promised by Vol. 6 Section 12, performance/load testing against Vol. 3's targets, the multi-tenancy and financial-integrity test suites that protect the platform's highest-severity risks, resilience/chaos testing for Vol. 5's integration failure modes, test data management, user acceptance criteria tied to Vol. 1's module scope, and the defect/release-gate process that plugs into Vol. 7's CI/CD pipeline.
1.2 Testing Philosophy
Given the modular-monolith-first decision (Vol. 2 Section 2.1) and the financial-integrity stakes (Vol. 1 Principle P4), this platform favors a test pyramid weighted toward fast, deterministic unit and integration tests, with a thin layer of end-to-end tests reserved for the value streams where cross-module correctness cannot be verified any other way (Section 6). Manual exploratory testing is retained deliberately for UI/UX judgment calls (Vol. 9) that automation cannot meaningfully assess.

4.1 Scope
Integration tests verify a module's interaction with its own database schema (Vol.4), its published/consumed domain events (Vol.2 Section 10), and its calls into other modules' public APIs (Vol.2 Section 11) - run against the ephemeral CI environment (Vol.7 Section 2.1) with real PostgreSQL, Kafka, and Redis instances (via containers), not mocks, since the highest-value bugs in this platform live at exactly these boundaries.

5. Contract Testing
5.1 API Contract Tests
Every public API endpoint (Vol.3 Section 5, Vol.2 Section 11) has a contract test asserting its request/response schema matches its published OpenAPI definition (Vol.3 Section 4.2 CS-05) - run in CI on every change to either the endpoint implementation or the schema, catching drift before it reaches an external integrator (Vol.5 Section 7).
5.2 Event Schema Contract Tests
Every domain event (Vol.2 Section 10 catalog) has a schema-registry-backed contract test (Vol.3 Section 4.2 CS-06): a producer cannot publish a payload that violates the registered schema, and a new schema version must be backward-compatible (additive only) unless explicitly declared as a new major version. This is what makes the event-driven architecture in Vol. 2 Section 2.2 safe to evolve without breaking every downstream consumer simultaneously.
5.3 Consumer-Driven Contract Verification
For the highest-fan-out events (JournalPosted, InvoiceSent - Vol.2 Section 10), each consumer module publishes its own expectations of the event shape it depends on; the producer's CI pipeline runs all registered consumer contracts before merge, catching a breaking producer change even if the consumer's own repository/module hasn't been touched.

9. Multi-Tenancy & Financial Data Integrity Testing
9.1 Multi-Tenancy Suite
Beyond the cross-tenant leakage suite (Section 7), this includes cache-key isolation tests (Vol.3 Section 7.2 tenant-prefixed keys), tenant-context-propagation tests (Vol.3 Section 11.2 - verifying every async job and event consumer correctly carries tenantId, not just synchronous API requests), and tenant-deletion/offboarding tests (Vol.4 Section 10) verifying a deactivated tenant's data is inaccessible without being prematurely purged.
9.2 Financial Integrity Suite
Double-entry balance invariant: after every test scenario (unit, integration, or E2E) that posts to the ledger, assert total debits equal total credits for every affected journal entry and, in aggregate, for the tenant's full ledger.
Append-only enforcement: attempt direct UPDATE/DELETE against ledger.journal_lines using the application's database role and assert it is rejected (Vol.4 Section 4.1 principle D2).
Reversal correctness: for every voided/reversed transaction, assert the reversing entry exactly negates the original and that both remain queryable (never physically deleted).
Idempotent replay: replay the same domain event twice through a consumer and assert no double-posting occurs (Vol.3 Section 8.2 idempotent-consumer requirement).
Audit hash-chain integrity: run the periodic verification job (Vol.6 Section 9.2) against a seeded audit log and assert it detects an artificially tampered entry.

12.3 UAT Sign-off Process
Each MVP acceptance criterion is verified against staging by a business stakeholder (or a designated proxy), not solely by the engineering team that built it.
A criterion is only marked accepted with a recorded pass note (screenshot or recorded session) attached to the criterion ID, ensuring an auditable UAT record exists.
Release 1 GA sign-off (Vol.7 Section 5.2 manual approval gate) requires 100% of MVP-tagged acceptance criteria accepted, with any exceptions explicitly waived in writing by the product owner.
